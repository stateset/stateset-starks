# StateSet STARK: Technical Whitepaper

## 1) Executive Summary

StateSet STARK is a Rust-native proof stack that provides:

1. Per-event compliance proofs (`ves-stark-air`, `ves-stark-prover`, `ves-stark-verifier`) that hide a private amount while proving it satisfies a policy.
2. Batch state-transition proofs (`ves-stark-batch`) that aggregate many per-event compliance decisions into a deterministic, hash-based state root transition.

The design goal is to keep compliance logic transparent to the verifier while preserving privacy of sensitive transaction amounts and enabling chainable batch commitments.

The implementation currently lives in `crates/ves-stark-air`, `crates/ves-stark-batch`, `crates/ves-stark-primitives`, `crates/ves-stark-client`, and companion verifier/prover crates.

## 2) Scope and Threat Model

The system proves two classes of statements:

1. A single-event statement: “there exists a private amount satisfying a policy under agreed parameters.”
2. A batch statement: “all events in this batch are processed correctly, merklized, and transform `(prev_state_root → new_state_root)` consistently.”

Security assumptions align with the repository threat and soundness notes in `docs/SOUNDNESS.md` and `docs/THREAT_MODEL.md`:
- Trust in STARK soundness under chosen `ProofOptions`.
- Correctness of STARK-friendly cryptographic primitives.
- Proper application-layer binding of witness commitments and policy metadata.
- Payload-to-amount linkage is intentionally outside the current AIR; protocols must enforce this if required.

## 3) High-Level Architecture

1. Public inputs are validated and normalized by `ves-stark-primitives` (`CompliancePublicInputs`).
2. Per-event witness is built and serialized as AIR columns by `ves-stark-prover`.
3. Batch witness is assembled, validated, and expanded into a multi-phase trace by `ves-stark-batch` (`BatchWitnessBuilder`, `BatchTraceBuilder`).
4. Proofs are generated by Winterfell-backed prover modules.
5. Verifiers in `ves-stark-verifier` and `ves-stark-batch` accept proofs only under approved parameter sets (`default`, `fast`, `secure`).
6. Optional HTTP integration and chain submission are handled in `ves-stark-client`.

## 4) Cryptographic Foundations

1. Field: Goldilocks prime, `p = 2^64 - 2^32 + 1`, via `ves-stark-primitives`.
2. Hash: Rescue-Prime (`STATE_WIDTH = 12`, `ROUNDS = 7`) with S-box exponent `α = 7`.
3. Commitments:
   - Per-event witness commitment: `rescue_hash(amount_limbs)[0..3]`.
   - Event leaf commitment includes event id, amount commitment, policy hash, and compliance bit, then Rescue-hashed.
4. Batch state root: `state_root = RescueHash(event_tree_root || metadata_hash)`.

## 5) Public Inputs and Canonicalization

1. Event-level public inputs are JSON-canonicalized and validated (`serde_jcs` pipeline in `ves-stark-primitives`).
2. Policy hash is domain-separated:
   `policy_hash = SHA256("STATESET_VES_COMPLIANCE_POLICY_HASH_V1" || JCS({ policyId, policyParams }))`.
3. `witness_commitment` in public inputs is optional for non-bound decoding, but verifiers strongly recommend hard-binding it.
4. Batch public inputs are fixed-width (`33` field elements) and include:
   - `prev_state_root`, `new_state_root`,
   - `batch_id`, `tenant_id`, `store_id`,
   - `sequence_start`, `sequence_end`, `num_events`,
   - `all_compliant`,
   - `policy_hash`, `policy_limit`.

## 6) Phase 1: Per-Event Compliance AIR

### 6.1 Intended Statement

Given public inputs and a public witness commitment `C`, prove there exists `amount` such that:
1. `amount` satisfies policy (`< threshold` or `<= cap`),
2. `C == Rescue(amount_limbs)`,
3. public inputs are correctly bound into the trace.

This is implemented in `crates/ves-stark-air/src/compliance.rs`.

### 6.2 Constraint Highlights

1. U64 amount represented as 8 × u32 limbs, with limbs 2..7 constrained to zero.
2. Bit decomposition enforced for limbs 0 and 1 (`32 + 32` bits).
3. Recomposition constraints rebuild limbs from bits.
4. Comparison gadget enforces subtraction-style bound and terminal borrow behavior.
5. Rescue constraints bind witness commitment from `RESCUE_STATE` evolution and output row.
6. Boundaries assert metadata, flags, round counters, and policy fields at fixed rows.

Reference: `NUM_CONSTRAINTS = 157` in `crates/ves-stark-air/src/compliance.rs`.

## 7) Phase 2: Batch AIR and Batch State Transition

Batch proofs are defined in `crates/ves-stark-batch/src/air/batch_air.rs`.

### 7.1 State Model

1. `EventLeaf` = `(event_id, amount_commitment, policy_hash, compliance_flag)` where `amount_commitment` is 4-element Rescue output.
2. `EventMerkleTree` computes a padded Rescue Merkle root.
3. `BatchStateRoot = RescueHash(event_tree_root || metadata_hash)`.
4. Metadata hash includes batch/tenant/store UUIDs and sequence/timestamp fields (`BatchMetadata::to_rescue_hash`).

### 7.2 Witness Validation Semantics

`BatchWitness::validate` enforces:

1. event count and non-empty constraints,
2. sequence continuity within batch (`sequence_number == sequence_start + index`),
3. unique `event_id`s,
4. policy consistency by ID/params/hash and limit across events,
5. tenant/store consistency,
6. witness-commitment correctness (recomputed and compared),
7. witness-compliance flag correctness.

### 7.3 Trace Composition

`BatchTraceBuilder` lays out three phases (`Event`, `Merkle`, `Finalize`, plus padding):

1. Event phase: `ROWS_PER_EVENT = 4` rows per event.
2. Merkle phase: `ROWS_PER_MERKLE_NODE = 15` per internal node (`14` transition + `1` output), over `2^ceil(log2(n)) - 1` internal nodes.
3. Finalize phase: `FINALIZE_ROWS = 15` rows for `state_root`.
4. Total trace length:
`next_power_of_two(event_rows + merkle_rows + finalize_rows)` with floor at `256`.

Formula from `crates/ves-stark-batch/src/air/trace_layout.rs`:
- `event_rows = num_events * 4`
- `merkle_rows = (2^ceil(log2(num_events)) - 1) * 15`
- `finalize_rows = 15`
- length rounded to power-of-two and at least `256`.

### 7.4 Compliance Accumulator

1. `COMPLIANCE_ACC_GAMMA = 7`.
2. Initialized to `GAMMA^{-num_events}` and multiplied by `gamma * flag` at one row per event.
3. Final accumulator equals `1` iff all flags are one; otherwise a reduced value, giving a non-trivial transition profile while retaining a clean boolean final value.

### 7.5 Constraint Budget

1. `NUM_BATCH_CONSTRAINTS = 54 + 47 + NUM_COMPLIANCE_BINDING_CONSTRAINTS`.
2. Merkle/finalization constraints = `47` (`crates/ves-stark-batch/src/air/constraints/merkle.rs`).
3. Compliance binding constraints currently = `102` (`NUM_COMPLIANCE_BINDING_CONSTRAINTS` in `crates/ves-stark-batch/src/air/constraints/compliance_binding.rs`).
4. Assertions count = `44`.

The compliance binding module enforces:
1. bit validity of amount limbs,
2. limb recomposition,
3. u64 upper-limb zero checks,
4. comparison value binary + monotonic behavior,
5. Rescue-state-to-amount binding,
6. final flag equality at row `EVENT_ROW == 2`.
All are gated by active-event selector (`EVENTS_DONE == 0`).

## 8) Verifier Semantics

### 8.1 Single Event Verification

1. Proof bytes and public inputs are syntax-checked.
2. Policy hash and policy identity/limit are re-derived and compared.
3. Witness commitment is verified against caller input; optional hard-fail if `public_inputs.witness_commitment` exists and mismatches.
4. Proof is accepted only under allowed options (`default`, `fast`, `secure`).
5. Result includes timing, policy id/limit, and validity status.

Implemented in `crates/ves-stark-verifier/src/verify.rs`.

### 8.2 Batch Verification and Chain Checks

`BatchVerifier` adds:

1. max proof-size enforcement (`10 MB`),
2. `all_compliant` must be field bit,
3. sequence range consistency (`sequence_end - sequence_start + 1 == num_events` with overflow checks),
4. per-proof validity,
5. optional chain checks:
   - continuity of state roots,
   - sequence continuity:
     `batch[i].sequence_start == batch[i-1].sequence_end + 1`.

## 9) Security Posture and Known Limitations

1. The implementation enforces policy and witness binding in-circuit for both per-event and batch aggregation.
2. Batch layer adds deterministic structural integrity (Merkle + finalization) and chain-level continuity checks.
3. Remaining critical limit (explicit in docs):
   - No in-AIR link between encrypted/plain payload hashes and the hidden amount.
   - A protocol must ensure event decoding and amount extraction integrity before generating public inputs.

## 10) API and Integration Points

1. Client flow for single events: `ComplianceProver` + `CompliancePublicInputs` -> proof -> `verify_compliance_proof_auto...`.
2. Batch flow:
   - `BatchWitnessBuilder` -> `BatchProver` -> proof bytes + batch metadata.
   - `BatchVerifier::verify_chain` for continuity and state integrity.
3. External submission support for Set Chain is in `ves-stark-client` (`batch_integration.rs`) with protocol-specific root and root-hash handling.

## 11) Current Hardening Status (Recent Updates)

1. Compliance binding in batch AIR strengthened from a minimal single check to full event-scope binding constraints (`102` constraints).
2. Sequence continuity checks added to batch chain verification (`prev.sequence_end + 1 == curr.sequence_start`).
3. Root serialization endianness clarified and made explicit (`to_hex_le`/`from_hex_le`) in `BatchStateRoot`.
4. Witness serialization helpers in client hardened with explicit little-endian byte conversion and consistency tests.

## 12) Conclusion

StateSet STARK combines privacy-preserving per-event policy proofs with a batched, hash-based state-transition system that is amenable to chain-of-trust integration. The practical security value is in the batch-level linkage and strict witness-policy consistency, while retaining a clear extension path for stronger payload-to-amount attestation inside the AIR.
